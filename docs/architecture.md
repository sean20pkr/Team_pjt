# Architecture

## 1. 문서 목적
- 이 문서는 `채널물량 인사이트 AI AGENT`가 어떤 구조로 동작하는지 쉽게 설명하는 문서입니다.

## 2. 서비스 한 줄 구조
- 이 서비스는 `숫자형 데이터 + 문서형 데이터 + 운영 메모`를 함께 읽고, `조회`, `예측`, `보고 보조` 결과를 만드는 구조입니다.

## 2-1. 설계 원칙
- 초기 구현은 `조회형 우선`, 이후 `예측형 추가`의 2단계 구조로 설계합니다.

## 3. 입력 계층
### 3-1. 숫자형 데이터
- 월별 업적 물량
- 조직 효율 숫자
- CSM
- 시장 규모
- 경쟁보험사 숫자 및 비교 지표

### 3-2. 문서형 데이터
- 월별 시책 세부내용
- 신상품/운영 공지
- 채널별 특이 이슈 메모

### 3-3. 운영 입력
- 담당자 월별 이슈 입력
- 기존 보고자료 업로드

## 3-4. 입력 표준화 원칙
- 사람이 입력할 수 있는 범위만 요구합니다.
- 입력 데이터 템플릿을 먼저 표준화합니다.
- 경쟁보험사 비교는 동일 기준으로 맞춘 데이터만 사용합니다.

## 3-5. mock 데이터 매핑 초안
- `main_fact.csv`: 핵심 사실 테이블 후보
- `monthly_summary.csv`: 월별 집계 테이블 후보
- `monthly_events.csv`: 월별 이벤트/시책/프로모션 테이블 후보
- `channel_profile.csv`: 채널 마스터 후보
- `product_profile.csv`: 상품 마스터 후보
- `special_products.csv`: 특수 상품 보조 테이블 후보
- `dummy_data_bundle.json`: 통합 전달 포맷 또는 API 응답 예시 후보
- `channel_volume_dummy_data.xlsx`: 현업 검토용 보기 자료 후보

## 4. 핵심 처리 계층
### 4-1. 조회 엔진
- 특정 월의 숫자와 관련 문서를 함께 묶어 보여줍니다.
- 판촉비, 프로모션, 경쟁사/시장 이슈를 시점 기준으로 빠르게 찾습니다.

### 4-2. 맥락 정리 엔진
- 시책, 신상품, 운영 이슈, 경쟁 흐름을 숫자와 연결해 설명합니다.

### 4-3. 예측 엔진
- 과거 패턴과 동반 변수들을 바탕으로 다음달 총계와 채널별 방향성을 추정합니다.
- 경쟁보험사 숫자와 상대 우세/열세 비교도 함께 해석합니다.

### 4-4. 보고 보조 엔진
- 사람이 바로 검토할 수 있는 요약 문장과 체크 포인트를 만듭니다.

### 4-5. 인사이트 엔진
- 과거 상황, 현재 이슈, 경쟁 흐름을 함께 읽고 타겟 물량 달성에 도움이 되는 시사점을 생성합니다.

## 5. 출력 계층
- 조회 답변
- 예측 답변
- 인사이트 답변
- 보고용 초안
- 추가 확인 포인트

## 6. 핵심 실행 흐름
1. 사용자가 월, 채널, 비교 대상, 질문 목적을 입력합니다.
2. 시스템이 관련 숫자와 문서를 함께 찾습니다.
3. 조회 질문이면 사실과 맥락을 함께 정리합니다.
4. 예측 질문이면 과거 흐름과 동반 변수를 기반으로 다음달 방향성을 계산합니다.
5. 결과를 숫자, 설명, 근거, 한계, 체크 포인트 형태로 반환합니다.
6. 사람은 검토 후 내부 판단과 공식 보고에 사용합니다.

## 7. 중요 원칙의 구조 반영
- AI는 보조 도구이며 최종 판단자는 사람이므로, 결과는 `확정안`이 아니라 `참고 인사이트`로 표시합니다.
- 예측 결과는 숫자만 주지 않고 근거와 한계를 같이 제공합니다.
- 데이터가 부족하면 억지로 확정 답변하지 않고 부족하다고 표시합니다.

## 8. 조회형 처리 기준
- 사실 조회는 빠르고 일관되게 나와야 합니다.
- 숫자와 관련 이슈가 함께 연결되어야 합니다.
- 경쟁보험사 비교는 동일 기준으로 비교되어야 합니다.

## 9. 예측형 처리 기준
- 총계 기준 숫자 예측이 가능하면 제시합니다.
- 전월 대비 방향성이 맞아야 합니다.
- 채널별 설명이 논리적으로 따라와야 합니다.
- 설명 없는 숫자 예측은 좋은 답변으로 보지 않습니다.

## 10. 사람 검토 지점
- 공식 보고 문장으로 전환하기 전
- 데이터 부족 경고가 포함된 답변일 때
- 채널별 방향성이 크거나 민감한 차이를 보일 때
- 경쟁보험사 비교 해석이 포함될 때

## 10-1. 운영 반영 지점
- 사람 승인 시점은 보고 반영 프로세스와 함께 정의합니다.
- 임원 보고용 1장 요약 자료로 전환되기 전에는 사람이 반드시 검토합니다.

## 11. 실패와 예외 처리
- 필요한 숫자 데이터가 없으면 누락으로 표시합니다.
- 관련 문서가 부족하면 근거 부족으로 표시합니다.
- 예측은 가능하지만 신뢰도가 낮으면 참고용임을 더 강하게 표시합니다.
- 숫자와 설명이 충돌하면 검토 필요 상태로 분류합니다.

## 11-1. mock 데이터 검증 흐름
1. `main_fact.csv`와 `monthly_summary.csv`로 검색 답변의 숫자 일치 여부를 확인합니다.
2. `monthly_events.csv`로 시책, 프로모션, 이슈 설명이 붙는지 확인합니다.
3. `channel_profile.csv`와 `product_profile.csv`로 채널/상품 기준 설명이 자연스러운지 확인합니다.
4. `special_products.csv`로 예외 상품이나 특수 케이스 설명이 가능한지 확인합니다.
5. `dummy_data_bundle.json`과 `channel_volume_dummy_data.xlsx`로 통합 응답과 현업 검토 형태를 확인합니다.

## 12. 같이 논의할 항목
- 예측 엔진의 출력에 신뢰도 등급을 넣을지
- 채널별 방향성 설명의 최소 근거 개수를 정할지
- 공식 보고용 문장과 내부 검토용 문장을 분리할지
- 경쟁보험사 우세/열세 비교를 어떤 공식 기준으로 고정할지
