# 삼성생명 물량예측 봇 화면 기획서

## 1. 화면 목적
- 사용자가 한 번의 질문으로 물량 조회, 설명, 전망·보고를 처리할 수 있게 한다.
- 결과는 답변만 보여주되, 필요한 경우 담당자 확인이 자연스럽게 이어지도록 설계한다.
- 화면은 모바일 우선을 기준으로 만들고, 데스크톱에서는 좌측 입력 / 우측 결과 구조로 확장한다.

## 2. 정보구조 IA
### 위에 둘 것
- 서비스 제목과 짧은 설명
- 면책 배너
- 질문 입력창
- 실행 버튼
- 현재 상태 배지

### 중간에 둘 것
- 답변 카드
- 담당자 확인 필요 배지
- 추가 확인 필요 회색 배지
- 근거 자료 영역

### 접을 것
- 상세 근거 문서 목록
- 월별 메타데이터
- 고급 설명 보조 문구
- 내부 운영 안내

### 읽는 순서
1. 제목과 면책
2. 질문 입력
3. 실행
4. 답변
5. 근거
6. 담당자 확인 여부

## 3. 핵심 흐름
1. 질문 입력
2. 실행 버튼 클릭
3. 로딩 상태 노출
4. 답변 + 근거 + 담당자 확인 여부 노출
5. 필요 시 추가 확인 또는 범위 밖 안내

## 4. 화면 상태 5종
### 4-1. 초기
- 기본 예시 질문 3개를 보여준다.
- 입력창은 비어 있지 않아도 되지만, 첫 진입 시 안내 문구가 보인다.
- 답변 영역에는 대기 상태가 표시된다.

### 4-2. 입력 중
- 사용자가 질문을 타이핑하면 하단에 입력 가이드가 유지된다.
- 개인정보처럼 보이는 숫자 패턴이 감지되면 실행 전에 경고한다.
- 실행 버튼은 활성 상태지만, 위험 패턴이 있으면 제출을 막는다.

### 4-3. 로딩
- 버튼은 `답변 생성 중...`처럼 바뀐다.
- 답변 카드에는 스켈레톤 또는 짧은 대기 문구를 보여준다.
- 화면 이동 없이 같은 영역에서 진행 상태를 보여준다.

### 4-4. 결과
- 답변 본문을 가장 먼저 보여준다.
- 그 아래에 근거 자료를 카드형으로 노출한다.
- 권한이 낮으면 근거 요청 안내 대신 상태 배지만 보여준다.
- `추가 확인 필요`가 있으면 회색 배지를 함께 표시한다.

### 4-5. 오류·범위 밖
- 질문이 범위를 벗어나면 답변 대신 범위 밖 안내를 보여준다.
- 서버 오류는 재시도 버튼과 함께 짧게 설명한다.
- 사용자가 다음 행동을 쉽게 알 수 있게 문구를 짧게 유지한다.

## 5. 컴포넌트
### 5-1. 질문 입력
- 여러 줄 텍스트 영역
- 예시 질문 버튼 3개
- 개인정보 유사 숫자 차단 안내

### 5-2. 실행 버튼
- 기본 문구: `질의 실행`
- 로딩 문구: `답변 생성 중...`
- 비활성은 최소화하고, 대신 검증 오류만 막는다.

### 5-3. 답변 카드
- 답변 본문을 가장 크게 표시한다.
- 상단에 상태 배지를 둔다.
- `전망·보고`는 보고형 문장처럼 한 문단으로 보여준다.

### 5-4. 근거 자료
- 근거를 2~4개 카드로 보여준다.
- 일반 사용자는 요약 근거만 보고, 슈퍼 유저는 상세 근거를 요청 후 본다.
- 근거는 접을 수 있는 구조로 둔다.

### 5-5. 담당자 확인 필요 배지
- 답변이 짧거나 근거가 부족할 때 노출한다.
- 회색 배지로만 표시하고, 과한 경고 문구는 쓰지 않는다.

### 5-6. 면책 배너
- 상단 고정 또는 카드 상단 배치
- 문구는 짧고 명확하게 유지
- 공식 판단을 대체하지 않는다는 점을 강조

## 6. UX 라이팅
### 버튼
- `질의 실행`
- `답변 생성 중...`
- `다시 질문`

### 플레이스홀더
- `예: 2026년 5월 전사 월초는 얼마야?`
- `예: 2026년 6월 전사 월초를 숫자로 확정할 수 있어?`

### 빈 상태
- `질문을 넣으면 답변과 근거를 보여드립니다.`

### 오류
- `질문을 처리하지 못했습니다. 다시 시도해주세요.`
- `질문 범위를 다시 써주세요. 조회, 설명, 전망·보고 중 하나로 이해하면 더 정확합니다.`

### 범위 밖
- `현재는 mock 데이터와 등록된 근거 범위 안에서만 답변합니다.`

### 개인정보 차단
- `전화번호나 주민번호처럼 보이는 숫자는 넣지 말아주세요.`

## 7. 접근성
- 대비비는 본문 4.5:1 이상을 목표로 한다.
- 모든 버튼은 키보드로 이동 가능해야 한다.
- 답변과 오류는 `aria-live`로 읽히게 한다.
- 모바일에서 터치 타겟은 44px 이상으로 둔다.
- 읽기 순서는 제목 → 입력 → 실행 → 결과 → 근거 순서를 유지한다.

## 8. 개인정보 숫자 차단 규칙
- 6자리 이상 연속 숫자는 우선 경고한다.
- 전화번호, 주민번호, 카드번호처럼 보이는 숫자 패턴은 제출을 막는다.
- 사업자번호처럼 보이는 숫자도 입력 시 경고 후 재확인하도록 한다.
- 이 화면은 개인 식별 숫자를 수집하는 용도가 아니다.

## 9. 화면 배치 원칙
- 상단에는 서비스 인식과 면책을 둔다.
- 입력창은 가장 먼저 보이게 한다.
- 결과는 답변 먼저, 근거는 그 다음에 둔다.
- 근거는 접고 펼 수 있게 한다.
- 모바일에서는 한 컬럼, 데스크톱에서는 좌우 2열로 확장한다.

## 10. 합의 기준
- 질문 입력 하나로 모든 핵심 기능이 작동해야 한다.
- 답변은 보고형 문장으로 보여주고, 숫자는 가능한 경우 최선의 추정과 근거를 함께 붙여야 한다.
- 답변 끝에는 실행 시사점 또는 보완 포인트를 짧게 붙인다.
- `담당자 확인 필요`는 조건이 맞을 때만 보여준다.
- 개인정보처럼 보이는 숫자는 입력 전에 막아야 한다.
